## 1. 这个滤波器是干什么的

滑动平均滤波器(Moving Average Filter)是一种**线性FIR滤波器**,对最近的N个采样求算术平均作为输出:

```
y[n] = (x[n] + x[n-1] + x[n-2] + ... + x[n-N+1]) / N
```

**为什么有效**: 独立同分布的高斯噪声,其均值趋于零。通过平均,噪声的方差降到原来的1/N:

```
Var(y) = Var(x) / N
```

**在嵌入式中的典型用途**:

- 平滑ADC采样(电源噪声、白噪声)
- 直流信号提取(传感器静态偏置校准)
- 周期噪声抑制(只要窗口长度等于噪声周期的整数倍)
- 过采样(Over-sampling)后的数字滤波

## 2. 滑动平均 vs 中值滤波 vs LPF

这三个模块覆盖了嵌入式滤波的主要场景:

| 滤波器 | 噪声类型 | 输出延迟 | 计算量 | 平滑效果 |
|--------|---------|---------|-------|---------|
| 滑动平均 | 高斯/白噪声 | (N-1)/2 | O(1) | 随N增大而增强 |
| 中值滤波 | 脉冲/椒盐噪声 | (N-1)/2 | O(N²) | 对脉冲完全抑制 |
| 一阶LPF | 任意噪声 | 时间常数τ | O(1) | 指数衰减加权 |

**选择指南**:

```
信号中是否有大幅度的异常跳变(毛刺)?
├── 是 → 用中值滤波
└── 否 → 需要什么样的频率响应?
    ├── 需要截至频率可调 → 用一阶LPF
    └── 需要简单均匀加权 → 用滑动平均
```

**滑动平均的特殊优势**:

1. **频率响应有固定零点**: 可以完全抑制特定频率(如50Hz工频)
2. **群延迟恒定**: 所有频率的延迟都是(N-1)/2个采样,没有LPF的相位失真
3. **线性相位**: 不会改变信号波形形状(只是整体偏移)

## 3. 结构体定义逐行拆解

```c
typedef struct {
    float *buffer;      //环形缓冲区指针
    uint16_t size;      //缓冲区大小
    uint16_t index;     //当前写入位置
    float   sum;        //窗口内数据和,用于快速求平均
    uint16_t count;     //已采样数量,未满时有效
} MovingAvg_t;
```

**逐行解释**:

| 成员 | 类型 | 含义 | 为什么需要 |
|------|------|------|-----------|
| `buffer` | float* | 环形缓冲区,存储最近N个采样值 | 维护滑动窗口数据 |
| `size` | uint16_t | 窗口大小(等于采样数N) | 确定平均分母 |
| `index` | uint16_t | 环形缓冲区的当前写入位置 | 实现FIFO覆盖 |
| `sum` | float | 窗口内所有数据的累加和 | **避免每次重新求和**(O(1)更新) |
| `count` | uint16_t | 已采样数(启动阶段<size) | 窗口未满时用count做分母 |

**关键设计亮点**: `sum` 的设计。

没有`sum`的朴素实现:

```c
// 不维护sum,每次重新求和
float sum = 0.0f;
for (int i = 0; i < size; i++) sum += buffer[i];
return sum / size;
```

维护`sum`的实现:

```c
sum = sum - oldest + newest;
return sum / size;
```

**前者是O(N),后者是O(1)**。对于N=10,差别不大;对于N=100或更高的过采样场景,**维护sum是性能的关键**。

**代价**: sum可能累积浮点误差。长期运行后,sum与实际每个元素之和之间会有微小差异。但float的精度对于N<1000通常不是问题。

## 4. MovingAvg_Init 逐行拆解

```c
uint8_t MovingAvg_Init(MovingAvg_t *ma, float *buf, uint16_t size)
{
    if (ma == NULL || buf == NULL || size == 0) return 1;
    ma->buffer = buf;
    ma->size   = size;
    ma->index  = 0;
    ma->sum    = 0.0f;
    ma->count  = 0;
    for (uint16_t i = 0; i < size; i++) {
        buf[i] = 0.0f;
    }
    return 0;
}
```

**逐行解释**:

- `if (ma == NULL || buf == NULL || size == 0) return 1;` — 三重空指针+零尺寸保护。size=0会导致除零错误,所以必须检查。

- `ma->buffer = buf;` `ma->size = size;` — 存储外部缓冲区指针和大小。

- `ma->index = 0;` — 从起始位置写入。

- `ma->sum = 0.0f;` — 累加和清零。

- `ma->count = 0;` — 采样计数归零。

- `for (...) { buf[i] = 0.0f; }` — 缓冲区清零。不这样做的话,sum(为0)与缓冲区实际内容(随机)不一致,第一次Update时`sum = sum - old + input`会对错误的old做操作。

## 5. MovingAvg_Reset 逐行拆解

```c
void MovingAvg_Reset(MovingAvg_t *ma)
{
    if (ma == NULL || ma->buffer == NULL) return;
    ma->index = 0;
    ma->sum   = 0.0f;
    ma->count = 0;
    for (uint16_t i = 0; i < ma->size; i++) {
        ma->buffer[i] = 0.0f;
    }
}
```

**逐行解释**:

与Init几乎一致,但不重置`size`(保留配置)。完全恢复到Init刚完成的状态。

## 6. MovingAvg_Update 逐行拆解

```c
float MovingAvg_Update(MovingAvg_t *ma, float input)
{
    if (ma == NULL || ma->buffer == NULL || ma->size == 0) {
        return input;
    }
```

**防御**: 任何无效状态都透传输入值,保证上层功能不中断。

```c
    float old = ma->buffer[ma->index];       //取出最旧数据用于更新累加和
```

**这是O(1)滑动平均的核心操作**:

`ma->buffer[ma->index]` 存放的是**当前将要被覆盖的、最旧的数据**。因为环形缓冲区中,index始终指向最早写入的那个元素(在窗口满的情况下)。

证明: 缓冲区初始全是0,index=0。第1次写入buffer[0],index变为1。第size次写入后,index回到0。此时buffer[0]恰好是最早写入的数据。

```c
    ma->buffer[ma->index] = input;
```

新数据覆盖旧数据。

```c
    ma->sum = ma->sum - old + input;         //减去最旧值加上新值,维持滑动和
```

**滑动求和的核心更新**:

```
sum_new = sum_old - buffer_oldest + input_new
```

- 减去最旧值: 把它从窗口中移除
- 加上新值: 把它加入窗口
- 结果: sum始终维持窗口内所有元素的和

**这个更新是O(1)的,不依赖于窗口大小N**。

```c
    ma->index++;
    if (ma->index >= ma->size) {             //环形缓冲区,回绕到起点
        ma->index = 0;
    }
```

指针后移,超出则回绕。

```c
    if (ma->count < ma->size) {              //窗口未填满时用当前数据量求平均
        ma->count++;
        return ma->sum / (float)ma->count;
    }
```

**窗口未满**: 启动阶段,count<size。此时窗口还未收到N个有效数据,用count作为分母。

例如: size=10, count=3, sum=前3个数据的和,输出 = sum/3。

```c
    return ma->sum / (float)ma->size;        //窗口填满后按固定窗口大小平均
}
```

**窗口已满**: 正常运行时,分母固定为size。

**注意**: 这里没有处理`ma->size == 0`的情况——虽然Init和Update入口处都检查了,但如果有人在运行中把size改成0(没有setter函数,直接写结构体成员),会导致除零。但这不是模块的问题,是使用者的责任。

## 7. 数学公式全解析

### 7.1 离散时间方程

滑动平均的差分方程(FIR形式):

```
        1   N-1
y[n] = ── ·  Σ  x[n - k]
        N   k=0
```

这是一个**N阶FIR滤波器**,所有系数都是1/N。

### 7.2 Z变换与频率响应

对差分方程做Z变换:

```
        1   1 - z^{-N}
H(z) = ── · ─────────
        N    1 - z^{-1}
```

频率响应(z = e^{jω}):

```
        1   sin(ωN/2)   -jω(N-1)/2
H(ω) = ── · ───────── · e
        N   sin(ω/2)
```

**幅频响应**:

```
        | 1   sin(ωN/2) |
|H(ω)| = |── · ─────────|
        | N   sin(ω/2)  |
```

这是一个**sinc-like函数**。关键特性:

1. **主瓣**: ω = 0时,H(0) = 1(直流增益=1,信号不失真)
2. **零点**: ω = 2πk/N 时( k=1,2,...,N-1 ),H(ω) = 0
3. **旁瓣**: 主瓣宽度 = 4π/N,旁瓣电平约 -13dB

### 7.3 零点频率与工频抑制

滑动平均最重要的特性是**在特定频率完全抑制信号**:

当窗口长度N满足: 零点频率 f_zero = f_s · k / N

取k=1,第一个零点在:

```
f_zero = f_s / N
```

**如果采样频率f_s=1000Hz,窗口N=20,则f_zero=50Hz**——完美抑制50Hz工频干扰!

这是滑动平均相比一阶LPF的重大优势: **LPF只能衰减特定频率,不能完全消除,而滑动平均在零点处增益是0**。

### 7.4 3dB截止频率

滑动平均的-3dB截止频率近似为:

```
f_c ≈ 0.443 · f_s / N
```

| N | 截止频率(归一化) | 举例(f_s=100Hz) |
|---|-----------------|----------------|
| 5 | 0.089·fs | 8.9 Hz |
| 10 | 0.044·fs | 4.4 Hz |
| 20 | 0.022·fs | 2.2 Hz |
| 50 | 0.009·fs | 0.9 Hz |

## 8. 参数选择指导

### 8.1 窗口大小N的选择

滑动平均的**唯一参数**就是窗口大小N。

**N的选择**: 需要理解三个trade-off:

| N | 平滑效果 | 延迟 | 截止频率 | 抑制零点 |
|---|---------|------|---------|---------|
| 小(3-5) | 弱 | 小 | 高 | 少 |
| 中(10-20) | 中等 | 中等 | 中等 | 可抑制工频 |
| 大(50-100) | 强 | 大 | 低 | 多 |

**典型选择**:

| 应用场景 | 推荐N | 理由 |
|---------|-------|------|
| ADC降噪(通用) | 8~16 | 窗口大小适中 |
| 50Hz工频抑制(f_s=1000) | 20 | 使f_s/N=50 |
| 60Hz工频抑制(f_s=1200) | 20 | 使f_s/N=60 |
| 过采样提高分辨率 | 4的幂次(4,16,64) | 配合过采样率 |
| 慢速传感器(温度) | 50~100 | 响应速度不重要 |
| 快速控制回路 | 3~5 | 最小化延迟 |

### 8.2 抑制特定频率

如果需要抑制特定频率f_noise:

```
N = round(f_s / f_noise)
```

**举例**: 采样率500Hz,需要抑制50Hz:

```
N = 500/50 = 10
```

这样50Hz及其谐波(100Hz,150Hz,...)都被零点抑制。

但这样做的**代价**: N=10时,截止频率约22Hz。如果信号在22Hz以上,也会被衰减。

### 8.3 窗口大小与延迟

滑动平均的延迟是固定的(线性相位):

```
延迟(采样数) = (N-1) / 2
延迟(时间)   = (N-1) / (2 · f_s)
```

**举例**:

| N | f_s=100Hz的延迟 | f_s=1000Hz的延迟 |
|---|----------------|-----------------|
| 3 | 10 ms | 1 ms |
| 10 | 45 ms | 4.5 ms |
| 20 | 95 ms | 9.5 ms |
| 50 | 245 ms | 24.5 ms |

## 9. 典型调用链

```
// 场景1: ADC过采样与滑动平均
void ADC_IRQHandler()
{
    uint16_t adc_val = ADC_Read();  // 硬件ADC读取
    float filtered = MovingAvg_Update(&ma, (float)adc_val);
    // 直接使用filtered值
}

// 场景2: 电流采样平滑(用于PID反馈)
void CurrentControlLoop()
{
    float raw_current = ReadCurrentSensor();
    float smooth_current = MovingAvg_Update(&ma_current, raw_current);
    pid_output = PID_Update(target_current, smooth_current);
    SetPWM(pid_output);
}

// 场景3: 温度传感器(慢速,大窗口)
void TemperatureTask()
{
    float raw_temp = ReadTempSensor();
    float avg_temp = MovingAvg_Update(&ma_temp, raw_temp);
    if (avg_temp > TEMP_THRESHOLD) {
        Fan_ON();
    }
}
```

**被谁调用**:

| 调用位置 | 处理数据 | 推荐窗口N |

| 温度采集任务 | 温度传感器 | 50~100 |
| 电压监测 | 电池电压 | 10~20 |

## 10. 踩坑点大全

### 10.1 坑1: 窗口边界处理不当

**现象**: 启动阶段的输出异常(远小于正常值)。

**根因**: 窗口未满时,count作为分母。如果信号正常值很大(如10000),前几个输出依次为10000、5000、3333、2500...直到N次后才稳定到10000。

这个**启动爬坡**是滑动平均的固有特性。

**解决方案**:

(1) 用第一个有效值预填充缓冲区:

```c
float first = ReadSensor();
MovingAvg_Init(&ma, buf, 10);
for (int i = 0; i < 10; i++) {
    MovingAvg_Update(&ma, first);  // N次快速填入
}
// 或者直接填充buffer
```

(2) 在Init时不清零,用第一个值填满:

```c
// 外部辅助函数
void MovingAvg_InitWithValue(MovingAvg_t *ma, float *buf, uint16_t size, float init_val)
{
    MovingAvg_Init(ma, buf, size);
    for (uint16_t i = 0; i < size; i++) {
        ma->buffer[i] = init_val;
    }
    ma->sum = init_val * size;
    ma->count = size;  // 强制标记为满
}
```

### 10.2 坑2: sum的浮点误差积累

**现象**: 长时间运行后,sum的值与buffer中各元素实际求和的结果不一致。

**根因**: sum的更新 `sum = sum - old + input` 中,每次涉及两次加减,float的舍入误差会累积。N=1000,运行10万次后,误差可能达到0.1%量级。

**解决方案**:

(1) 周期性重新计算sum:

```c
// 每1000次调用重新计算一次sum,消除累积误差
static uint32_t call_count = 0;
call_count++;
if (call_count % 1000 == 0) {
    float recal_sum = 0.0f;
    for (uint16_t i = 0; i < ma->size; i++) {
        recal_sum += ma->buffer[i];
    }
    ma->sum = recal_sum;
}
```

(2) 对大多数嵌入式应用,误差在可接受范围内,不需要处理。

### 10.3 坑3: 窗口N的奇偶选择

**与中值滤波不同,滑动平均不要求N为奇数**。偶数N同样有效。

偶数N的一个有趣特性: 零点在 f_s/2, f_s/4, f_s/6... 奇数N的零点在 f_s/3, f_s/5, f_s/7...

根据你要抑制的噪声频率选择对应的零点分布。

### 10.4 坑4: sum的管理与复位一致性

**现象**: Reset后忘记重置sum,导致输出异常大或异常小。

**分析**: Reset将sum置0,但buffer归零,此时sum和buffer是一致的(都是0)。但如果Reset不清空buffer只重置sum,就会出问题。

**还好**: 这个模块的Reset既清空了buffer也重置了sum,没有这个bug。

### 10.5 坑5: 信号中混有脉冲噪声

**滑动平均对脉冲噪声很敏感**: 一个大幅度的脉冲会导致连续N个输出都受到影响(因为脉冲值要在窗口内停留N次)。

**对比**: 中值滤波只需一次就能消除脉冲。如果信号中既有高斯噪声又有脉冲噪声:

```c
float median = MedianFilter_Update(&mf, raw);
float avg = MovingAvg_Update(&ma, median);  // 先去脉冲,再平滑
```

### 10.6 坑6: 实时更新时buffer[index]的陷阱

```c
float old = ma->buffer[ma->index];  // 在更新index之前取old
ma->buffer[ma->index] = input;
ma->sum = ma->sum - old + input;
ma->index++;
```

**顺序至关重要**: 先取old再写新值,old才是真正被替换的那个。如果先写了新值再取old,取到的就是刚刚写进去的input——相当于减去同一个值,sum不变,滤波完全失效。

## 11. 初始化与启动瞬态

### 11.1 启动阶段输出变化

假设N=5,输入为恒定值100:

| 调用次数 | buffer内容 | count | sum | 输出 |
|---------|-----------|-------|-----|------|
| 0(Init) | [0,0,0,0,0] | 0 | 0 | - |
| 1 | [100,0,0,0,0] | 1 | 100 | 100/1=100 |
| 2 | [100,100,0,0,0] | 2 | 200 | 200/2=100 |
| 3 | [100,100,100,0,0] | 3 | 300 | 300/3=100 |
| 4 | [100,100,100,100,0] | 4 | 400 | 400/4=100 |
| 5 | [100,100,100,100,100] | 5 | 500 | 500/5=100 |

如果初始值是0,输入是100,从第一次起输出就是100——看起来没有启动瞬态。

**但当初始值和第一个输入值不一致时**:

| 调用次数 | buffer内容 | 输出 |
|---------|-----------|------|
| 0(Init) | [0,0,0,0,0] | - |
| 1 | [100,0,0,0,0] | 100 |
| 2 | [100,200,0,0,0] | 150 |
| 3 | [100,200,300,0,0] | 200 |
| 4 | [100,200,300,400,0] | 250 |
| 5 | [100,200,300,400,500] | 300 |

**这才是真实的启动瞬态**: 如果信号从0跳到100,输出不会立即到100,而是逐渐爬升。N越大,爬升越慢。

### 11.2 信号突变时的响应

如果信号从稳定值100突变到200:

```
旧窗口: [100, 100, 100, 100, 100] → 输出 100
第1次: [200, 100, 100, 100, 100] → 输出 600/5=120
第2次: [200, 200, 100, 100, 100] → 输出 700/5=140
第3次: [200, 200, 200, 100, 100] → 输出 800/5=160
第4次: [200, 200, 200, 200, 100] → 输出 900/5=180
第5次: [200, 200, 200, 200, 200] → 输出 1000/5=200
```

**所以**: 从100到200的阶跃,需要N次采样才能完全跟踪。这是滑动平均的"斜坡响应"——线性相位滤波器的特性。

## 12. 调试手段

### 12.1 串口打印对比

```c
float raw = ReadSensor();
float avg = MovingAvg_Update(&ma, raw);
printf("%lu,%f,%f,%f\n", micros(), raw, avg, ma.sum);
```

**观察要点**:
- 噪声幅度: 原始曲线vs滤波后曲线的噪声幅度对比,应该是 1/sqrt(N)
- 延迟: 阶跃响应时,滤波后曲线应滞后 (N-1)/2 个采样
- sum: 观察sum的变化是否平滑,如果出现异常跳动,说明有错误

### 12.2 阶跃响应测试

输入已知的阶跃信号(如从100→200),测量滤波输出的10%~90%上升时间:

```
理论上: 上升时间 ≈ (N-1)/2 个采样
实际: 应与理论一致,否则说明有bug
```

### 12.3 频率响应验证

如果带有DAC,可以产生正弦信号,用ADC采集并通过滑动平均滤波:

- 低频输入(远低于f_c): 输出与输入几乎一致,无衰减
- 接近f_c: 输出衰减约3dB

- 高频: 输出被大幅衰减

## 13. 为什么都用float

与其他模块一致。补充说明滑动平均的特殊性:

滑动平均中`sum`变量是**累加和**,对于N=100、信号范围0~4095(12位ADC):

```
最大sum = 100 × 4095 = 409500
```

这个值远小于float的最大精度损失范围(约16777216以内精确表示整数),所以用float精确无误。

如果用uint32_t做sum:
- 没问题,但要注意ADC值转换为float,求和再转回float,多一次转换
- 而且如果N很大(如1000),sum可能溢出(uint32_t最大42亿)

如果用int32_t:
- 有符号32位最大21亿,对于ADC+大窗口有风险
- 且需要处理负数

**结论**: float最省心,在MSPM0G3507的FPU支持下性能无损。

## 14. 为什么用环形缓冲区

### 14.1 朴素实现(不用环形缓冲区)

如果不用环形缓冲区,每次新数据来时需要搬移数据:

```c
// 朴素实现: 每次搬移
for (int i = size-1; i > 0; i--) {
    buffer[i] = buffer[i-1];  // 所有数据右移一位
}
buffer[0] = new_data;  // 新数据放最前面
```

代价: 每次Update要搬移N-1个float,复杂度O(N)。

### 14.2 环形缓冲区实现

```c
// 环形缓冲区: 不搬移,只用一个指针
buffer[index] = new_data;
index = (index + 1) % size;
```

代价: O(1)。缓冲区不需要搬移任何数据,只需更新指针。

### 14.3 代价

环形缓冲区的代价是**缓冲区中的元素不是按时间顺序排列的**。如果要遍历窗口内所有数据(如调试用),需要:

```c
// 从最旧到最新遍历
for (uint16_t i = 0; i < ma->size; i++) {
    uint16_t idx = (ma->index + i) % ma->size;  // 从当前index开始
    printf("%f,", ma->buffer[idx]);
}
```

但这不影响滤波功能,因为滑动平均只关心sum,不关心数据的顺序。

## 15. 为什么维护sum变量

### 15.1 不使用sum的朴素版本

```c
float MovingAvg_Update_Naive(MovingAvg_t *ma, float input)
{
    ma->buffer[ma->index] = input;
    ma->index = (ma->index + 1) % ma->size;

    float sum = 0.0f;
    for (uint16_t i = 0; i < ma->size; i++) {
        sum += ma->buffer[i];
    }
    return sum / (float)ma->size;
}
```

**计算量**: 每次求和N次加法 + N次数组访问 = 2N个操作。

### 15.2 使用sum的优化版本(当前实现)

```c
float MovingAvg_Update(MovingAvg_t *ma, float input)
{
    float old = ma->buffer[ma->index];
    ma->buffer[ma->index] = input;
    ma->sum = ma->sum - old + input;     // 只做2次加减
    ma->index = (ma->index + 1) % ma->size;
    return ma->sum / (float)ma->size;
}
```

**计算量**: 2次加减 + 2次数组访问 = 4个操作。

### 15.3 性能对比

| N | 不加sum(每次全量计算) | 维护sum |
|---|---------------------|--------|
| 5 | 10次操作 | 4次操作 |
| 10 | 20次操作 | 4次操作 |
| 50 | 100次操作 | 4次操作 |
| 200 | 400次操作 | 4次操作 |

**维护sum后,性能与窗口大小无关**。对于大窗口(如过采样16倍,N=64),不使用sum的版本性能差16倍。

### 15.4 维护sum的唯一代价

**浮点误差积累**: 前面讲过,这是唯一的代价。但从工程角度看:

- float的精度约7位有效数字
- sum的误差积累速度约 1e-7 × N × 调用次数
- 对于N=20,调用100万次,误差约 2e-2(相对0.002%)

这个误差完全可接受。

---

> **总结**: 滑动平均是最简单的FIR滤波器,O(1)的更新算法使其在大窗口下保持高效,独特的频率零点使其可以精确抑制特定频率噪声。在嵌入式系统中,它是"平均去噪"场景的不二之选。
